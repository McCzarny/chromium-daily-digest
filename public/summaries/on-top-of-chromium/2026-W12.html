Week 12 of 2026 (2026-03-15 to 2026-03-21)

Chromium Weekly: 2026 Week 12

Overview

This week saw an exceptionally high volume of changes, with 1732 relevant commits out of 2328 total. Development was dominated by a wave of breaking API changes as part of the ongoing 'Bedrock' project to abstract core browser interfaces. Major policy shifts were also enacted, including enabling Device Bound Session Credentials by default and altering Certificate Transparency enforcement. For developers building on Chromium, this week requires careful review to address numerous required code migrations.

Core APIs and Interfaces: Widespread Breaking Changes

  • ⚠️ BREAKING BREAKING CHANGE As part of the 'Bedrock' refactoring project, many functions like chrome::FindTabbedBrowser and chrome::FindBrowserWithWindow now return the abstract BrowserWindowInterface* instead of a concrete Browser*. This is a fundamental change requiring updates across any code that interacts with the browser window. (4a045d1) (ba815fd) (4b4f769) (84b2352)
  • ⚠️ BREAKING BREAKING CHANGE Several core interfaces now require new method implementations, which will break existing subclasses. content::WebContents added RequestSelectionBounds, content::SecurityPrincipal added IsWebUI, and syncer::DataTypeSyncBridge added TrimAllSupportedFieldsFromRemoteSpecifics. (8020df7) (b7f87db) (a531d36)
  • ⚠️ BREAKING BREAKING CHANGE On Windows, the native event observation API in base/message_loop/message_pump_win.h was refactored. Will/DidDispatchMSG methods were renamed to Will/DidRunNativeEvent and their signature changed to take a uintptr_t identifier instead of const MSG&. (fe915f0) (6be9b03)
  • ⚠️ BREAKING BREAKING CHANGE The core International Components for Unicode (ICU) library was upgraded to version 78.2. This may cause subtle behavioral changes in internationalization APIs. A related fix was landed to ensure HTTP date formatting continues to use the literal 'GMT' string, as required by RFC 7231, instead of the new 'GMT+0' default. (f72d48e) (81cd430)
  • ⚠️ BREAKING BREAKING CHANGE The media::VideoFrame release callback signature has changed. It now receives an additional bool lost_resource parameter to indicate if the underlying shared image was lost, requiring updates for all custom video producers. (e614ea8)
  • ⚠️ BREAKING BREAKING CHANGE The IdentityManager initialization flow was simplified by removing the account_consistency parameter from its builder, requiring updates to embedder initialization code. (63307d4)

Security

  • Device Bound Session Credentials (DBSC) are now enabled by default, and the associated origin trials have been removed. This enhances security against cookie theft by changing the default session registration behavior. (ebf37aa)
  • ⚠️ BREAKING BREAKING CHANGE A major Certificate Transparency (CT) policy change now causes Chromium to ignore Signed Certificate Timestamps (SCTs) delivered via OCSP responses by default. Websites relying solely on this mechanism for CT compliance may now trigger certificate trust errors. (4d1a2f7)
  • ⚠️ BREAKING BREAKING CHANGE IDNA 2008 is now enabled by default for URL processing. This is a significant behavioral change to URL canonicalization that may affect how internationalized domain names are parsed and handled. (3ae9984)
  • A critical bug in Windows registry handling was fixed, preventing the improper closing of handles to predefined keys like HKEY_CURRENT_USER, which could break command-line registry overrides. (b2a5c41)
  • IPC validation was hardened for WindowOpenDisposition values sent from renderers, and validation was added for headers on renderer-initiated navigations to mitigate exploits from compromised processes. (4ccc2d9) (fdab935)

Extensions

  • ⚠️ BREAKING BREAKING CHANGE Nine methods were removed from the chrome.passwordsPrivate extension API, including deleteAllPasswordManagerData and isAccountStorageActive. Extensions must migrate to the new password_manager.mojom interface for password management functionality. (17f18ce)
  • ⚠️ BREAKING BREAKING CHANGE The ServiceWorkerState::Observer::OnWorkerStop method signature now includes a blink::ServiceWorkerToken, requiring embedders observing extension service workers to update their observer implementations. (2cc7179)
  • A new performance-oriented feature flag, kOptimizeWebRequestProxy, was introduced. When enabled, it avoids globally proxying all subresource requests through the browser process just because a <webview>-using extension is installed, enabling more direct renderer-to-network requests. (95c9427)
  • The extension updater gained "session hunting" logic to retry failed downloads from the Chrome Web Store by incrementing the authuser query parameter, resolving authentication issues when multiple Google accounts are signed in. (56bfb92)

Browser Features and UI

  • ⚠️ BREAKING BREAKING CHANGE The Tab Organization feature and its associated TabOrganizationService have been completely removed from the codebase. Any integrations with this service will no longer compile. (4a44e72)
  • A new method, DetachWebContents(tabs::TabHandle tab), was added to the TabListInterface, allowing for the transfer of a tab's WebContents ownership to the caller for use in other UI surfaces. (81e7070)
  • A new method, CreatedBySessionRestore(), was added to the BrowserWindowInterface, enabling components to identify windows that were created as part of a session restore operation. (f200ba1)
  • On macOS, the First Run experience has been fully migrated from a native Cocoa implementation to the cross-platform Views framework, unifying the UI across platforms. (5ce0c2b)

Networking and Performance

  • DNS-over-HTTPS (DoH) requests will now use an empty IsolationInfo, which allows Alt-Svc information to be persisted across requests and enables faster adoption of QUIC on subsequent connections. (6ca275c)
  • A new PRESUBMIT check was added to disallow new usages of base::RunLoop in //chrome, encouraging developers to adopt patterns that are less prone to re-entrancy bugs. (d44615c)
  • ⚠️ BREAKING BREAKING CHANGE On Windows, the EME API will now report HDCP 2.3 as unsupported for PlayReady-protected content, correcting previous behavior where it was treated identically to HDCP 2.2. This may affect content providers who relied on the previous reporting. (87eee40)
